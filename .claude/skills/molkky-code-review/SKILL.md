# モルックスコアラー コードレビューチェックリスト

実装完了後、PRを作成する前にこのチェックリストで自己検証する。過去に発生した重大バグの教訓をベースに構成。

## 1. 文字化け・エンコーディング

過去の教訓: Unicode文字化けが3,436箇所発生した事故、Phase 5でのUnicode文字化け283箇所+絵文字破壊18個の事故がある。

- [ ] 日本語テキストがUnicodeエスケープシーケンス(`\u524a\u9664`等)になっていないか → 直接日本語で書く
- [ ] スマートクォート(`'` `'` `"` `"`)が混入していないか → ASCII引用符(`'` `"`)のみ使用
- [ ] emダッシュ(`—`)やUnicode省略記号(`…`)が混入していないか → `--`や`...`を使用
- [ ] 絵文字が意図せず破壊されていないか（特にバルク置換後）

## 2. Import/Export整合性

過去の教訓: ファイル分割時にimport漏れが複数発生。

- [ ] 新しく追加した関数・コンポーネントがexportされているか
- [ ] 削除・移動した関数のimport元が全ファイルで更新されているか
- [ ] 使われていないimportが残っていないか
- [ ] 循環importが発生していないか

## 3. ShuffleAnimation安全性

過去の教訓: ShuffleAnimationバグで全画面ブラックアウトが発生。

- [ ] ShuffleAnimationコンポーネントのreturn全体がtry-catchでラップされているか
- [ ] stall時の安全弁（自動クローズタイマー）が機能しているか
- [ ] onDoneコールバックが確実に呼ばれるパスになっているか（未呼び出しバグの前科あり）
- [ ] skipConfirm stateが正しく管理されているか

## 4. ビルド検証

- [ ] `npx vite build` がエラーなしで完了するか
- [ ] コンソールにwarningが出ていないか（特にReact key、未使用変数）
- [ ] console.logが残っていないか（console.errorは残してよい）

## 5. PWA互換性

- [ ] iOS Safari PWAとして正常に動作するか（特にフルスクリーン、スクロール）
- [ ] IndexedDBの読み書きが正常か
- [ ] オフライン時にクラッシュしないか（ネットワークエラーのハンドリング）

## 6. 既存機能への影響

- [ ] ゲームロジック(reducer, adv, scoreOf, failsOf, getPI)に変更を加えていないか
- [ ] 既存のスタッツ計算ロジックに影響を与えていないか
- [ ] Supabase同期が正常に動作するか
- [ ] 管理者/メンバー権限の区別が維持されているか
- [ ] マルチコート機能（iPad端末コート+紙コート）が正常か

## 7. UI視認性（GameScreen変更時のみ）

- [ ] アクティブ/非アクティブチームの区別が色だけでなくレイアウトでも明確か
- [ ] 3m離れて読めるフォントサイズか
- [ ] 入力ボタンサイズがisTabletロジック上限を超えていないか
- [ ] 4チーム×4人の最大構成でレイアウトが崩れないか
- [ ] 長いチーム名・プレイヤー名で表示が壊れないか

## 8. セキュリティ

- [ ] APIキーやシークレットがクライアントサイドコードに露出していないか
- [ ] 新しいAPIエンドポイントにPIN認証が必要な場合、実装されているか
- [ ] CORSヘッダーが適切に設定されているか

## 9. コミット品質

- [ ] コミットメッセージが日本語で書かれているか
- [ ] 1コミットの変更範囲が適切か（1つの論理的変更 = 1コミット）
- [ ] 関係ない変更（フォーマット変更、リファクタリング）が混入していないか

## 10. 正典化強度に応じた追加検証

PR 作成前のセルフレビューにおいて、変更対象画面・領域の **正典化強度** に
応じた追加検証を実施する。本章は spec-writer 側「指示書作成前のチェック」と
対称構造を取り、「書いた後の防御線」を担う。

### 10-1. 正典化強度マップ（参照）

正典化強度の現行マップは `.claude/skills/molkky-ui-design/SKILL.md` の
「画面別の設計方針」セクションに記載されている。本章では強度値を直接
記載せず、ui-design 側のマップを参照する形にする（二重メンテを避ける
ため）。

検証時は以下の手順で強度を確認する。

1. 変更対象の画面・領域を特定する
2. ui-design/SKILL.md の「画面別の設計方針」で当該画面の強度を確認
3. 強度に応じて 10-2 〜 10-4 のいずれかの追加検証を実施

ui-design 側のマップが更新された場合、本章は自動的に追従する（参照のみ
のため）。

### 10-2. 濃い領域に変更を加えた場合（必須）

変更対象画面の正典化強度が「濃い」の場合、PR本文に **改善エビデンス**
が3要素揃って記載されているかを検証する。

#### 必須3要素

1. **実機検証結果**: iPad Pro M5 等の実機で確認した具体的な問題点
2. **ユーザー観察**: 試合中・運用中に発生した実害（誰が・いつ・どう困ったか）
3. **既存実装が不十分である根拠**: 現行実装で解決できない理由

#### 検証手順

- PR本文を読み、上記3要素がすべて記載されているかを確認
- 1つでも欠ける場合、レビューを差し戻す（マージブロック相当）
- 3要素すべて揃っていても、内容が「実機検証ではなく推測」「ユーザー観察
  ではなく仮説」等の場合は、補強を求める

#### 該当画面（参考）

- GameScreen 本体（GameWindWidget を除く）
- WindMonitor

### 10-3. 中強度領域に変更を加えた場合

変更対象画面の正典化強度が「中強度」の場合、PR本文または直近の
コミットメッセージに **改善理由** が明記されているかを確認する。

#### 確認事項

- 改善理由が1〜2行でも明記されているか
- 改善理由が「なんとなく良くしたい」等の曖昧表現になっていないか
- 既存実装の「何が不十分か」が言語化されているか

改善理由がない・曖昧な場合、レビューコメントで質問する（マージブロック
ではなく、補足要請）。

#### 該当画面（参考）

- 試合カード
- 試合結果画面
- Settings
- SetupScreen
- Stats（方針 = 中強度）

### 10-4. 薄い領域に変更を加えた場合

変更対象画面の正典化強度が「薄い」の場合、通常レビュー基準で進める。

- エビデンス・改善理由は必須ではない
- ただしテスト・型チェック・ビルド検証等の機械的検証は通常通り実施
- AI 生成っぽい UI 回避チェックリスト（ui-design 側）等の標準チェックは
  従来通り適用

#### 該当画面（参考）

- Stats（具体実装）
- GameScreen の GameWindWidget（pill型風速計）

### 10-5. 三層ドメイン分類との連動

正典化強度は、三層ドメイン分類（ui-design 側）における **ドメイン1（現状実装）**
内での判定軸である。

- ドメイン1（現状実装）への変更: 強度判定が必須（10-2 〜 10-4 を適用）
- ドメイン2（拡張仕様）への新規追加: 暫定強度=薄い から始まる（10-4 適用、
  ただし 10-6 の暫定強度ルールも参照）
- ドメイン3（領域外）の変更: 基本不採用（PR 自体がレビュー対象外）

PR 作成前にドメイン分類を確認し、ドメイン1 への変更時は強度判定を、
ドメイン2 への新規追加時は暫定強度の明示を、それぞれ PR本文に記載する。

### 10-6. 新機能の暫定強度ルール

新機能（ドメイン2: 拡張仕様）として実装された変更は、最初は **暫定強度=薄い**
で扱う。

- 初回 PR では 10-4（薄い領域の検証）のみ適用
- 数試合運用して問題なければ、強度を中強度〜濃いに昇格させて確定する
- 暫定強度から確定強度への昇格は **太一さんの判断** で決まる
- 昇格判断は ui-design/SKILL.md「画面別の設計方針」の更新と連動する

#### 検証時の注意

- 新機能 PR では「これは暫定強度=薄い扱い」と PR本文に明示することを推奨
- 暫定強度のまま長期運用される場合、確定昇格の機会を逸する可能性がある
  （Claude Code 側からは昇格を提案できるが、最終判断は太一さん）

### 10-7. 強度判定が困難なケース

以下のケースでは、Claude Code 単独で強度判定をせず、太一さんに確認を求める。

- ui-design 側のマップに該当画面の記載がない（マップ更新漏れの可能性）
- 複数画面に跨る変更で、画面ごとに強度が異なる
- 「ドメイン1 と 2 の境界」で判定が割れる（既存画面に新要素を追加する等）

確認時は「該当画面・該当変更内容・判定が割れる理由」を明記して報告する。
