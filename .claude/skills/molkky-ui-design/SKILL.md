# モルックスコアラー UI設計ガイドライン

このスキルは、モルックスコアラーのUI作業（新規画面作成、既存UIの変更、デザインレビュー）で参照する。CLAUDE.mdの基本ルールに加えて、ここに詳細な設計思想と判断基準を記載する。

## 運用背景

このスキルは、モルックスコアラー固有の UI 設計思想を Claude Code が
参照するための運用ルールである。frontend-design プラグインの一般的な
方向性を、モルック固有の制約・運用環境・正典化された設計判断で上書きする。

中核となる思想は **機能とデザインの両立** である。屋外運用の視認性
（iPad 三脚固定 / 直射日光下 / 試合中の短時間判断）と、デザイン品質
（情報階層・余白・色数制御・装飾抑制）を、どちらかを犠牲にせず両立させる
ことを目指す。一方を諦めた瞬間に AI生成っぽい UI に堕する。

本スキルは UI v3 世代の正典化を反映している。各画面は「正典化強度」と
「質感アイデンティティ」を持ち、強度に応じて改修時の根拠強度を変える。
詳細は本ドキュメントの「画面別の設計方針」セクションを参照。

## 機能とデザインの両立思想

モルックスコアラーの UI 設計は、**機能とデザインのどちらかを犠牲にしない**
ことを中核思想とする。一方を諦めた瞬間に AI生成っぽい UI に堕する。

### 両立すべき2軸

- **機能軸**: 屋外運用の視認性・iPad 三脚固定下の操作性・試合中の短時間
  判断・誤操作回避
- **デザイン軸**: 情報階層の明瞭さ・余白と整列の品位・色数制御・装飾抑制・
  質感アイデンティティの一貫性

### 画面別の重み配分

すべての画面で両軸を満たすことを前提としつつ、画面の用途に応じて
**最も求められる要素 × 高いデザイン性の共存** を画面別方針として定める。

- GameScreen: **視認性 × デザイン**
- Stats: **分析性 × デザイン**
- Settings: **操作性 × デザイン**
- WindMonitor: **視認性 × デザイン**（航海計器風の濃い質感）
- SetupScreen: **操作性 × デザイン**（計器盤のブランド刻印）
- 試合結果画面: **振り返り体験 × デザイン**（Stats抑制版）
- 試合カード: **親画面同調 × デザイン**

各画面の質感アイデンティティの詳細は次セクション「画面別の設計方針」を参照。

### 両立を破る代表的アンチパターン

- 視認性のために色数を増やしすぎ、結果として AI生成っぽい UI に堕する
- デザインのために情報密度を下げすぎ、試合中の短時間判断ができなくなる
- 機能要件を理由に装飾過剰を許容し、情報階層が崩れる
- デザイン優先で機能色（警告色等）を弱め、誤操作リスクが上がる

これらは「どちらかを諦めた」結果であり、本スキルでは原則として許容しない。

## 画面別の設計方針

各画面は「正典化強度」と「質感アイデンティティ」を持つ。改修時は
正典化強度に応じて根拠強度を変える。

正典化強度の3段階:

- **濃い**: 既に正典化済みの確定領域。改修にはエビデンス（実機検証 +
  ユーザー観察 + 既存実装が不十分である根拠）が必須
- **中強度**: 改修方針は確定済みだが、具体実装は柔軟に許容する領域。
  改修理由の明記が必須
- **薄い**: 通常運用領域。エビデンス不要、通常のレビュー基準で進める

### GameScreen（正典化強度: 濃い、ただし GameWindWidget は薄い）

質感アイデンティティ: **試合中の短時間判断を支える計器盤**。
チーム情報ブロック（チーム名 + プレイヤー名 + 点数 + ミス）が凝集単位。
直射日光下視認性は装備で解決済みのため第一優先ではないが、
対策リソースは一定残す。

設計上の留意点:

- 情報ブロックを分割配置せず、まとまりとして表示する
- チーム固有色は廃止し、アクティブ/非アクティブの2状態のみで区別
- 背景はセミライト、濃紺はヘッダー・アクティブ強調に限定
- 入力部ボタンサイズは現行 isTablet ロジックの上限を超えない
- GameWindWidget（pill型風速計）のみ正典化強度=薄い扱い。
  通常のレビュー基準で改修可能

### Stats（方針=中強度、具体実装=薄い）

質感アイデンティティ: **濃厚質感 × Strava/Whoop 的ストイックさ**。
分析画面としての密度感と、試合後の落ち着いた振り返り体験を両立する。

設計上の留意点:

- 分析性とデザイン性を共存させる
- データの密度感を表現しつつ、試合直後の高揚を残さない抑制された質感
- 具体実装の試行錯誤は許容(強度=薄い)

### SetupScreen（正典化強度: 中強度）

質感アイデンティティ: **計器盤のブランド刻印**。試合開始前の準備時間を、
プロフェッショナルな計器を起動する儀式として演出する。

設計上の留意点:

- 操作性を最優先しつつ、設定画面としての計器感を表現
- ブランド要素（タイトル・バージョン情報等）の刻印感を保つ

### Settings（正典化強度: 中強度）

質感アイデンティティ: **操作性 × デザイン**。設定変更の確実性と、
誤操作回避を両立する。

設計上の留意点:

- 操作性を最優先しつつデザイン性も維持
- 変更が即座に反映される項目と確認を要する項目の区別を明確化

### WindMonitor（正典化強度: 濃い）

質感アイデンティティ: **航海計器風ダークテーマ**。風向風速の精密表示を
専用画面として独立させ、計器としての完結性を持つ。

設計上の留意点:

- 航海計器の質感（ダークベース、計器盤の精度感）を保持
- 改修にはエビデンス必須。色の意味の重複化を絶対回避

### 試合結果画面（正典化強度: 中強度）

質感アイデンティティ: **Stats抑制版の濃厚質感**。試合直後の振り返りに
ふさわしい、Stats の濃厚さを少し抑えた抑制された表現。

設計上の留意点:

- Stats と同系統の質感だが、より抑制された表現
- 試合直後の感情の波を増幅しすぎない設計

### 試合カード（正典化強度: 中強度）

質感アイデンティティ: **親画面に同調する控えめな器**。Stats画面・
試合結果画面に配置されるカード単体は、親画面の質感を損なわない控えめな
表現にとどめる。

設計上の留意点:

- 親画面の質感アイデンティティを優先する
- カード単体で主張せず、リスト全体・親画面全体での視認体験を優先

## 情報ブロックの概念

チーム名 + プレイヤー名 + 得点 + ミス回数を「意味のあるまとまり」として表示する。バラバラに散らばった個別数値ではなく、1つのブロックとして認識できるレイアウトにする。「大きな数字を1つ置く」より「情報ブロック全体がパッと認識できる」方が実戦では有用。

## 視線誘導の原則

- **主軸**: レイアウト（配置）・位置・サイズで視線を誘導する
- **補助**: 色のコントラストは補強手段であり、主手段ではない
- **品位の表現手段**: 余白、整列、情報階層、色数制御
- **禁止**: 中間色・薄いグラデーション・半透明・弱いボーダーで品位を出そうとしない

## 色の意味の重複化禁止

色の使い分けは UI の判別性に直結する。1つの色に複数の意味を持たせると、
ユーザーは「この色が何を意味するか」を文脈ごとに判断する必要が生じ、
試合中の短時間判断が破綻する。

### 色の役割カテゴリ

- **ブランド色**: アプリのアイデンティティを示す色（例: 濃紺）
- **アクション色**: ユーザー操作のフィードバックを示す色
- **状態色**: アクティブ/非アクティブ、選択状態等を示す色
- **警告色**: 誤操作回避・注意喚起を示す色（例: 赤系）
- **機能色**: 特定の機能要素（例: 風速、距離等）を識別する色

これらのカテゴリ間で色を共有させない。

### 正典化強度との関係

色の役割分離は、画面の正典化強度に応じて適用厳格度が変わる。

- **濃い領域（GameScreen本体 / WindMonitor）**: 絶対適用。
  1色1意味の原則を破る実装は受け付けない
- **中強度領域**: 強く推奨。やむを得ず重複させる場合は改修理由を
  コミットメッセージに明記
- **薄い領域**: 推奨。実装の柔軟性を許容しつつ、レビュー時に確認

### 代表的な重複化アンチパターン

- アクション色と警告色を同じ赤系で表現し、ユーザーが「タップしてよい」のか
  「警告」なのか判断できない
- 状態色（アクティブ）とブランド色を同じ濃紺で表現し、どこがアクティブで
  どこが装飾的な濃紺かが識別できない
- 機能色（風速のレベル表示）が他の状態色と被り、風速計か状態表示かが
  読み取れない

これらは v3 世代では明示的に避ける。色の追加は「削る」発想で慎重に行う。

### 確認手順

- 新規実装・改修時、画面内で使われる色を列挙する
- 各色がどのカテゴリ（ブランド/アクション/状態/警告/機能）に属するかを明示
- 同じ色が2カテゴリ以上に属していないかを確認
- 属している場合、正典化強度を確認し、濃い領域では実装を見直す

## frontend-design プラグインとの関係

frontend-design プラグインは Anthropic が提供する公式プラグインで、
一般的な Web UI 設計のベストプラクティスを Claude Code に提供する。
本スキル `molkky-ui-design` は、このプラグインの提示物を **モルック
スコアラー固有の制約で上書きする位置づけ** にある。

### 採否判断のロジック

frontend-design プラグインが提示するパターン・コンポーネント・スタイルが
モルックスコアラーで採用可能かを、以下の順で判定する。

#### Step 1: 機能とデザインの両立を破らないか

提示物が「機能軸」または「デザイン軸」のどちらかを犠牲にしていないかを
確認する。屋外運用の視認性・試合中の短時間判断・iPad 三脚固定の操作性
のいずれかを損なう提示は不採用。

#### Step 2: 該当画面の質感アイデンティティと整合するか

提示物が該当画面の質感アイデンティティ（GameScreen=計器盤、Stats=濃厚質感×
ストイック、WindMonitor=航海計器風、SetupScreen=ブランド刻印 等）と
整合するかを確認する。整合しない提示は不採用。

#### Step 3: 色の意味の重複化禁止に抵触しないか

提示物の配色が、画面内で既に役割を持つ色と重複していないかを確認する。
重複がある場合、特に正典化強度=濃い領域では即不採用。

#### Step 4: 正典化強度と改修根拠が釣り合うか

提示物が改修対象とする画面の正典化強度を確認する。

- 濃い領域: エビデンス（実機検証 + ユーザー観察 + 既存実装が不十分である
  根拠）が揃わない限り不採用
- 中強度領域: 改修理由が明確であれば採用
- 薄い領域: 通常レビュー基準で採用判断

### プラグイン提示物の良い使い方

- **アイデア源として参照する**: 「こういうパターンもある」という気付きを
  得る用途では積極的に使う
- **iPad 改善案の素材にする**: iPad 改善の採用チェックリストを通したうえで
  採否を決める
- **コンポーネント設計の出発点**: ゼロから設計するより、プラグイン提示を
  起点に「モルックでは何を変えるべきか」を考える方が早い

### プラグイン提示物の悪い使い方

- **そのまま採用する**: 「frontend-design がこう言ったから」だけでは
  採用根拠にならない
- **モルック固有制約を後回しにする**: 視認性・三脚固定・短時間判断を
  「あとで調整する」前提でプラグイン提示を採用するのは禁止
- **正典化強度の高い領域に無根拠で適用する**: 濃い領域への変更は
  プラグイン提示を採用根拠にできない

### 階層関係の明文化

- frontend-design プラグイン: 一般原則の提供
- `molkky-ui-design` スキル: モルック固有制約による上書き（本スキル）
- 個別 PR の改修判断: 上記2階層を踏まえた具体判断

判断が衝突した場合、**基本的には `molkky-ui-design` の記述を優先** する。

ただし以下の例外がある:

- 太一さんから明示的に「frontend-design プラグインの提示を試験的に
  導入する」旨の指示がある場合は、例外的にプラグイン提示の優先度を
  上げて採用検討する
- この例外運用は、外部のデザイン知見を取り込むための試験導入として
  位置づけられる。例外採用が指示される場合、太一さん側で導入根拠
  （どの提示物をなぜ取り込みたいか）が必ず明示される
- 例外採用された変更は、運用評価を経て本スキルへの正典化（中強度〜
  濃い領域への昇格 / 既存原則の改訂）の検討対象になる

Claude Code 自身の判断で frontend-design プラグインの優先度を上げる
ことは禁止。例外運用の起点は **必ず太一さんからの明示的指示** である
ことを徹底する。

## AI生成っぽいUI回避チェックリスト

UIを作成・変更した後、以下の5項目でセルフチェックする。

### ① 優先順位にメリハリがあるか
- その画面の「主役」が1つに絞られているか
- 全部が均一に整っている＝AI生成っぽい。意図的な不均一さを入れる
- 情報の順序: 最初に見る → 次に比較する → 最後に行動する

### ② テンプレ寄せ集めでなく、文脈から設計しているか
- UIパターンを機械的に当てはめていないか
- モルックの試合運用という「体験の本質」から導かれたレイアウトか
- このアプリならではの情報単位（情報ブロック）を先に決めてからUIパターンを選んでいるか

### ③ 色に役割とルールがあるか
- ブランド色/操作色/状態色/注意色が分離されているか
- 同じ色に複数の意味を持たせていないか
- 「綺麗かどうか」ではなく「迷わないかどうか」で判断する

### ④ 実データで検証しているか
- 長いチーム名(例:「フジサンロクニオウムナク」)で崩れないか
- 0件、異常値(50点ちょうど、3ミスで失格直前)で表示が壊れないか
- 4チーム×4人の最大構成で画面が溢れないか

### ⑤ 削ぎ落としができているか
- 情報・色・装飾が増えすぎていないか
- なくてもよい線・面・要素が残っていないか
- 迷ったら足すより削る。直射日光環境では「削る意識」が特に重要

## iPad改善の採用チェックリスト

以下のチェック項目を全て満たさない限り、変更は採用しない。

### #1. 直射日光下視認性は対策装備で解決済みであることを前提とする

直射日光下視認性は、最高級アンチグレアフィルム + 開放カバー + 遮光日傘 +
熱対策の装備セットで解決済みである。そのため UI 改善の採用判断において
「直射日光下視認性」を第一優先のチェック項目として扱わない。

ただし対策リソースは一定残す:

- 装備が万全でない状況（出先での試合・装備忘れ等）でも最低限の視認性を
  確保する
- 装備による解決と UI による解決の役割分担を明確に保つ（UI 側で過剰に
  視認性に振らない）
- 装備が将来的に変更される可能性を考慮し、極端な視認性軽視は避ける

採用判定の優先軸は v3 世代では以下の通り:

- 第一: その画面で最も求められる要素 × 高いデザイン性の共存
  （GameScreen=視認性×デザイン、Stats=分析性×デザイン 等）
- 第二: 質感アイデンティティとの整合
- 第三: 色の意味の重複化禁止の遵守
- 第四: 装備が万全でない状況での最低限の視認性

### #2. 3m離れてアクティブチームが一目で識別できるか

### #3. チーム名・現在プレイヤー名・得点・ミス回数が情報ブロックとして見えるか

### #4. 3チーム・4チーム構成でもレイアウトが崩れないか

### #5. 機能とデザインの両立を破っていないか（一方を諦めていないか）

## デザイン品質の具体ルール

- 透過処理した背景（カジノキャラ背景等）は「無いもの」として扱う
- モック画像作成時は可能なら内部でコードを実行し、実際の表示環境を再現する（文字数に応じた動的サイズ変更がコード仕様通りに表示されるかを実機相当で検証する）
- 入力部ボタンサイズは現行isTabletロジック上限を超えない

## 三層ドメイン分類

UI 改修・機能追加の要望が来た時、それが「現状実装の修正」「拡張仕様の受け入れ」
「領域外の不採用」のどれに該当するかを最初に判定する。判定を経ずに改修を
進めると、本来不採用にすべき要望が混入したり、既に正典化された領域が無根拠で
改変されるリスクがある。

### ドメイン1: 現状実装

既に正典化済みの確定領域を指す。具体的には正典化強度=濃い〜中強度の領域に
おける、現行実装そのもの。

#### 該当例

- GameScreen 本体の情報ブロック構造（チーム名+プレイヤー名+点数+ミス）
- WindMonitor の航海計器風ダークテーマ
- SetupScreen の計器盤ブランド刻印
- Stats の濃厚質感×ストイック方針
- 試合結果画面の Stats抑制版質感
- 試合カードの親画面同調設計

#### 改修方針

- 触らないのが基本
- 触る場合は正典化強度に応じた根拠強度が必要
  - 濃い領域: エビデンス（実機検証 + ユーザー観察 + 既存実装が不十分である
    根拠）必須
  - 中強度領域: 改修理由の明記必須
- 「ついでの修正」「リファクタリングの巻き添え」での改変は禁止

### ドメイン2: 拡張仕様

新機能の受け入れ枠。本アプリのスコープ内で、まだ実装されていないが将来的に
受け入れる可能性がある領域。

#### 該当例

- 新しい統計指標の追加（Stats 内）
- 新しい設定項目の追加（Settings 内）
- 新しい風速関連表示の追加（WindMonitor 内）
- マルチコート機能の拡張
- AI 分析機能の拡張（Phase 4 環境データ連携 等）

#### 改修方針

- 受け入れ可能
- 受け入れ時は `DESIGN.md` 該当章への追記で対応する
- 新機能は最初 **正典化強度=薄い** でスタートする
- 数試合運用して問題なければ強度を上げて確定する
- 確定後は「現状実装」ドメインに昇格する

### ドメイン3: 領域外

本アプリのスコープ外。基本的に不採用とすべき要望。

#### 該当例

- 他のスポーツへの汎用化（フィンランド発祥のモルック専用アプリである前提）
- SNS 連携の高度化（基本機能優先）
- ゲーム要素の追加（実用ツールとしての位置づけを優先）
- アプリ内通貨・課金システム
- 他アプリとの過剰な連携

#### 改修方針

- 基本不採用
- 太一さんから明示的に「領域外だが導入したい」指示がある場合のみ例外検討
- Claude Code 自身の判断で領域外の機能を追加することは禁止

### 判定フロー

要望が来た時、以下の順で判定する。

1. **ドメイン3（領域外）か?** → Yes なら基本不採用、太一さんに確認
2. **ドメイン2（拡張仕様）か?** → Yes なら DESIGN.md への追記要否を確認した
   うえで実装着手
3. **ドメイン1（現状実装）か?** → Yes なら正典化強度を確認し、必要な根拠
   強度を満たしてから実装着手

### 判定が難しいケース

- **ドメイン2 と 1 の境界**: 既存画面に新要素を追加する場合、既存要素を
  改変するか・既存に追加するだけかで判定が変わる
- **ドメイン3 と 2 の境界**: 「拡張」と「領域外」の判定は太一さんの判断に
  委ねる。Claude Code は判定が難しい場合、独自判断で進めず確認を求める

判定根拠は PR 本文に明記する（特にドメイン1 への変更時は正典化強度と
根拠強度を併記）。

## 人間の視点での最終検証

1. **人間の目にどう見えるか**: コードの合理性ではなく、デバイス画面に表示された時に人間の目にどう映るかを想像する
2. **屋外iPadで3m離れて見ることを想像する**: 室内モニターでの「見える」と屋外での「読める」は全く別物
3. **技術的最適解より人間にとっての最適解**: 動的フォントサイズ計算が「均一充填率」を達成しても、結果が小さすぎて読めないなら文字数別固定サイズが正解
4. **迷ったら太一さんに判断を仰ぐ**: モックや選択肢を提示してプレビューで検証。技術的都合だけで判断しない
