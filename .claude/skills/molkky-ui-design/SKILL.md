# モルックスコアラー UI設計ガイドライン

このスキルは、モルックスコアラーのUI作業（新規画面作成、既存UIの変更、デザインレビュー）で参照する。CLAUDE.mdの基本ルールに加えて、ここに詳細な設計思想と判断基準を記載する。

## 運用背景

モルックは時間制限のある屋外競技。iPad を三脚に固定し、直射日光下で1.5m高さから最大3mの距離で8-10人が同時に見る。操作者と周囲のプレイヤーは1ターンの数秒間で以下を同時判断する必要がある：

- 相手チームのスコア（スコア表から読み取り）
- 50点までの残り点数
- 相手の上がり目（あと何点でフィニッシュか）
- スコア推移から相手の調子を読む
- ミス回数と失格リスク

「読める」だけでは不十分。「一瞬で正しく状況把握できる」情報設計が必須。

## 画面別の設計方針

### GameScreen（試合中）— 視認性が絶対最優先
- 情報の優先順位: 現在のゲーム番号/ターン → アクティブチーム → 現在のプレイヤー → 得点 → ミス回数
- デザインの美しさは視認性を損なわない範囲でのみ追求
- 3m離れた複数人が直射日光下で即座に状況を把握できることが最終基準

### Stats画面 — おしゃれさとデザインセンス
- 試合後にゆっくり見る画面なので、デザイン品質を追求してよい
- 「プロのデザイナーが作ったような、AIっぽさのない高品質」が目標
- ただしユーザーが操作に迷わないことは大前提

### SetupScreen — 操作のしやすさ
- 試合前の素早い設定が目的
- ボタン・入力の直感性を最優先
- 視覚的な華やかさは不要

## 情報ブロックの概念

チーム名 + プレイヤー名 + 得点 + ミス回数を「意味のあるまとまり」として表示する。バラバラに散らばった個別数値ではなく、1つのブロックとして認識できるレイアウトにする。「大きな数字を1つ置く」より「情報ブロック全体がパッと認識できる」方が実戦では有用。

## 視線誘導の原則

- **主軸**: レイアウト（配置）・位置・サイズで視線を誘導する
- **補助**: 色のコントラストは補強手段であり、主手段ではない
- **品位の表現手段**: 余白、整列、情報階層、色数制御
- **禁止**: 中間色・薄いグラデーション・半透明・弱いボーダーで品位を出そうとしない

## frontend-design プラグインとの関係

frontend-designプラグインは「ボールド・印象的・個性的」方向に誘導するが、モルックスコアラーでは以下の点で上書きする：

- **フォント**: 個性的なフォントより、可読性の高いフォント。屋外3mで読めることが最優先
- **アニメーション**: 試合画面では情報伝達を妨げるアニメーションは禁止。スタッツ画面では演出可
- **レイアウト**: 非対称・グリッドブレイク・オーバーラップは試合画面では禁止。情報の即時認識が最優先
- **色**: 5系統(濃紺/黄色/グレー/白/黒)に制限。華やかなカラーパレットより「迷わなさ」を優先
- **背景**: 大気感のある背景効果よりも、高コントラスト可読性

Stats画面やShuffleAnimationなど、視認性制約が緩い画面ではfrontend-designの方向性を活かしてよい。

## AI生成っぽいUI回避チェックリスト

UIを作成・変更した後、以下の5項目でセルフチェックする。

### ① 優先順位にメリハリがあるか
- その画面の「主役」が1つに絞られているか
- 全部が均一に整っている＝AI生成っぽい。意図的な不均一さを入れる
- 情報の順序: 最初に見る → 次に比較する → 最後に行動する

### ② テンプレ寄せ集めでなく、文脈から設計しているか
- UIパターンを機械的に当てはめていないか
- モルックの試合運用という「体験の本質」から導かれたレイアウトか
- このアプリならではの情報単位（情報ブロック）を先に決めてからUIパターンを選んでいるか

### ③ 色に役割とルールがあるか
- ブランド色/操作色/状態色/注意色が分離されているか
- 同じ色に複数の意味を持たせていないか
- 「綺麗かどうか」ではなく「迷わないかどうか」で判断する

### ④ 実データで検証しているか
- 長いチーム名(例:「フジサンロクニオウムナク」)で崩れないか
- 0件、異常値(50点ちょうど、3ミスで失格直前)で表示が壊れないか
- 4チーム×4人の最大構成で画面が溢れないか

### ⑤ 削ぎ落としができているか
- 情報・色・装飾が増えすぎていないか
- なくてもよい線・面・要素が残っていないか
- 迷ったら足すより削る。直射日光環境では「削る意識」が特に重要

## iPad改善の採用チェックリスト

以下5つを全て満たさない限り、変更は採用しない：
1. 直射日光でコントラスト低下時にも主要情報が読めるか
2. 3m離れてアクティブチームが一目で識別できるか
3. チーム名・現在プレイヤー名・得点・ミス回数が情報ブロックとして見えるか
4. 3チーム・4チーム構成でもレイアウトが崩れないか
5. 美しさのために視認性を犠牲にしていないか

## デザイン品質の具体ルール

- 透過処理した背景（カジノキャラ背景等）は「無いもの」として扱う
- モック画像作成時は可能なら内部でコードを実行し、実際の表示環境を再現する（文字数に応じた動的サイズ変更がコード仕様通りに表示されるかを実機相当で検証する）
- 入力部ボタンサイズは現行isTabletロジック上限を超えない

## 人間の視点での最終検証

1. **人間の目にどう見えるか**: コードの合理性ではなく、デバイス画面に表示された時に人間の目にどう映るかを想像する
2. **屋外iPadで3m離れて見ることを想像する**: 室内モニターでの「見える」と屋外での「読める」は全く別物
3. **技術的最適解より人間にとっての最適解**: 動的フォントサイズ計算が「均一充填率」を達成しても、結果が小さすぎて読めないなら文字数別固定サイズが正解
4. **迷ったら太一さんに判断を仰ぐ**: モックや選択肢を提示してプレビューで検証。技術的都合だけで判断しない
