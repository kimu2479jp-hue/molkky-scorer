# Claude Code 設計指示書 作成ガイドライン

Claude AIとClaude Codeはメモリを共有していない。このスキルは、Claude AI側で設計した仕様をClaude Codeに正確に伝えるための指示書を作成する際のルールを定義する。

## 指示書の大原則

### テキスト完結の原則
- 画像への参照を一切含めない（「画像のように」「先ほどのスクショの通り」等は禁止）
- 過去の会話の暗黙的文脈を前提としない（「前に話した件」「例のバグ」等は禁止）
- 第三者（＝Claude Code）がこの指示書だけで、完全に作業を理解・実行できる形にする
- 再現手順・UI記述・バグ説明はすべて操作と結果をテキストだけで完結させる

### 段階的分割の原則
- 大規模変更は複数フェーズに分割する
- 各フェーズは独立してビルド・検証可能な単位にする
- フェーズ間の依存関係を明記する

## 指示書のテンプレート構造

```markdown
# [タスク名]: [簡潔な説明]

## 【Claude Code共通ルール】
- 確認事項、質問、コミットメッセージ、PRの説明文は全て日本語で表現すること
- 英語でのやり取りは禁止。全てのコミュニケーションを日本語で行うこと
- 不明点がある場合は日本語で質問すること
- 作業開始前に `git checkout main && git pull` でmainを最新にしてからブランチを作成

## 【鉄則】
- 求められていないリファクタリング・改善・フォーマット変更はしない
- 日本語テキストは直接日本語文字で記述する。Unicodeエスケープシーケンス禁止

## ブランチ
feature/[ブランチ名]

## 背景・目的
[なぜこの変更が必要か。ユーザー体験の観点から説明]

## 作業概要
[全体の作業を1-2文で要約]

## 作業1: [具体的な作業名]
### 変更対象ファイル
[ファイルパスを列挙]

### 変更内容
[具体的に何をどう変えるか。変更前→変更後の形式で書く]

### 完了条件
[この作業が「できた」と判断する基準]

## 作業完了後
- ブランチ名を報告
- PRは手動で作成するので不要
```

## バグ報告の書き方（BEERフレームワーク）

バグ修正を指示する場合は、以下の4要素を必ず含める：

- **B**ehavior expected（期待される動作）: 何が起きるべきか
- **E**rror observed（実際の動作）: 何が起きているか
- **E**nvironment（環境）: ブラウザ、デバイス、どの画面か
- **R**eproduction（再現手順）: バグを再現する具体的な操作手順

悪い例: 「ボタンが動かない」
良い例: 「SetupScreen.jsxの『ゲーム開始』ボタンをタップしても画面遷移しない。期待動作：GameScreenに遷移する。環境：iPad Safari PWA。再現手順：2チーム×2人を設定→ランダムモード選択→『ゲーム開始』タップ。console.errorにエラーなし。」

## UI変更指示の書き方

UI変更を指示する場合、以下を必ずテキストで記述する：

- **現在の状態**: 今どう表示されているか（色、サイズ、位置、テキスト）
- **変更後の状態**: どう表示されるべきか（具体的なCSSプロパティ値、色コード、px値）
- **影響範囲**: この変更で他の画面・コンポーネントに影響が出るか
- **検証方法**: 変更が正しいと確認する手順（どの画面で何を確認するか）

## 計画→承認→実装フロー

大規模な変更の場合、指示書の冒頭に以下を記載する：

```
【段階1】まず計画を教えてください。コードは書かないで。
- どのファイルを変更するか
- 技術的アプローチの選択肢（最もシンプルなものから）
- 想定されるリスク

私が承認したら実装に移ります。
```

## よくある失敗パターンの回避

- CLAUDE.mdが古い場合がある。指示書の冒頭で「現在の最新仕様」を明記する
- 「○○のように」という曖昧表現をすべて具体値に置き換えてから提出する
- 1フェーズ1チャットの原則を維持。長いチャットはClaude Codeの精度を下げる
