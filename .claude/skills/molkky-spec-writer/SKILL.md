# Claude Code 設計指示書 作成ガイドライン

Claude AIとClaude Codeはメモリを共有していない。このスキルは、Claude AI側で設計した仕様をClaude Codeに正確に伝えるための指示書を作成する際のルールを定義する。

## 指示書の大原則

### テキスト完結の原則
- 画像への参照を一切含めない（「画像のように」「先ほどのスクショの通り」等は禁止）
- 過去の会話の暗黙的文脈を前提としない（「前に話した件」「例のバグ」等は禁止）
- 第三者（＝Claude Code）がこの指示書だけで、完全に作業を理解・実行できる形にする
- 再現手順・UI記述・バグ説明はすべて操作と結果をテキストだけで完結させる

### 段階的分割の原則
- 大規模変更は複数フェーズに分割する
- 各フェーズは独立してビルド・検証可能な単位にする
- フェーズ間の依存関係を明記する

## 指示書作成前のチェック

指示書を書き始める前に、触る対象の正典化強度・三層ドメイン分類を確認する。
判断基準の詳細は `molkky-ui-design/SKILL.md` の「画面別の設計方針」
「三層ドメイン分類」セクションを参照すること。

このチェックを怠ると、Claude Code が「触ってよい領域」と
「改善エビデンスが必須の領域」を区別できず、既存資産が
意図せず改変されるリスクがある。

### Step 1: 三層ドメイン判定（触るかどうかの一次判定）

触る対象が以下のいずれに該当するかを最初に判定する。

- **現状実装**: 既に正典化済みの確定領域
  → 触らないか、触るなら厳格な根拠が必要
- **拡張仕様**: 新機能受け入れ枠
  → 受け入れ可能、DESIGN.md 該当章への追記で対応
- **領域外**: 本アプリのスコープ外
  → そもそも触らない（基本不採用）

### Step 2: 正典化強度判定（触る場合の根拠強度）

触ることが決まった場合、対象画面の正典化強度に応じて
指示書に含めるべき情報の濃度を変える。

#### 濃い強度（GameScreen 風速W除く / WindMonitor）

- 指示書の背景セクションに **改善エビデンス項目を必須記載**
- エビデンスの内容:
  - 実機検証結果（iPad Pro M5 で確認した具体的な問題点）
  - ユーザー観察（試合中に発生した実害）
  - なぜ既存実装が不十分かの根拠
- PR本文にも同じエビデンスを記載するよう指示書末尾で要求する
  （`molkky-code-review` スキル第10章と連動）

#### 中強度（試合カード / 試合結果画面 / 設定画面 / SetupScreen / Stats方針）

- 指示書の背景セクションに **改善理由を明記**
- コミットメッセージに改善理由を含めるよう Claude Code に指示する

#### 薄い強度（Stats実装 / GameScreen 風速W）

- 通常運用、特別な追加要求なし
- 段階的分割と検証手順は通常通り適用

### Step 3: 新機能の場合の暫定強度

新機能は最初**正典化強度=薄い**でスタートする扱いを指示書冒頭に記載。
数試合運用して問題なければ強度を上げて確定する旨も併記する。
これにより Claude Code 側でも「初回実装は確定版ではない」と
理解した上で実装してもらえる。

### チェック実施タイミング

- 指示書のドラフト作成**前**（=最初）に実施する
- 「背景・目的」セクションを書く時点ではすでに判定済みの状態にする
- 判定結果は指示書本文に明示しなくてよいが、
  Claude AI（指示書作成者）の判断根拠としては必須

## 指示書のテンプレート構造

```markdown
# [タスク名]: [簡潔な説明]

## 【Claude Code共通ルール】
- 確認事項、質問、コミットメッセージ、PRの説明文は全て日本語で表現すること
- 英語でのやり取りは禁止。全てのコミュニケーションを日本語で行うこと
- 不明点がある場合は日本語で質問すること
- 作業開始前に `git checkout main && git pull` でmainを最新にしてからブランチを作成

## 【鉄則】
- 求められていないリファクタリング・改善・フォーマット変更はしない
- 日本語テキストは直接日本語文字で記述する。Unicodeエスケープシーケンス禁止

## ブランチ
feature/[ブランチ名]

## 背景・目的
[なぜこの変更が必要か。ユーザー体験の観点から説明]

## 作業概要
[全体の作業を1-2文で要約]

## 作業1: [具体的な作業名]
### 変更対象ファイル
[ファイルパスを列挙]

### 変更内容
[具体的に何をどう変えるか。変更前→変更後の形式で書く]

### 完了条件
[この作業が「できた」と判断する基準]

## 作業完了後
- ブランチ名を報告
- PRは手動で作成するので不要
```

## バグ報告の書き方（BEERフレームワーク）

バグ修正を指示する場合は、以下の4要素を必ず含める：

- **B**ehavior expected（期待される動作）: 何が起きるべきか
- **E**rror observed（実際の動作）: 何が起きているか
- **E**nvironment（環境）: ブラウザ、デバイス、どの画面か
- **R**eproduction（再現手順）: バグを再現する具体的な操作手順

悪い例: 「ボタンが動かない」
良い例: 「SetupScreen.jsxの『ゲーム開始』ボタンをタップしても画面遷移しない。期待動作：GameScreenに遷移する。環境：iPad Safari PWA。再現手順：2チーム×2人を設定→ランダムモード選択→『ゲーム開始』タップ。console.errorにエラーなし。」

## UI変更指示の書き方

UI変更を指示する場合、以下を必ずテキストで記述する：

- **現在の状態**: 今どう表示されているか（色、サイズ、位置、テキスト）
- **変更後の状態**: どう表示されるべきか（具体的なCSSプロパティ値、色コード、px値）
- **影響範囲**: この変更で他の画面・コンポーネントに影響が出るか
- **検証方法**: 変更が正しいと確認する手順（どの画面で何を確認するか）

## 計画→承認→実装フロー

大規模な変更の場合、指示書の冒頭に以下を記載する：

```
【段階1】まず計画を教えてください。コードは書かないで。
- どのファイルを変更するか
- 技術的アプローチの選択肢（最もシンプルなものから）
- 想定されるリスク

私が承認したら実装に移ります。
```

## よくある失敗パターンの回避

### 1. CLAUDE.md の最新仕様明記

CLAUDE.md が古い場合がある。指示書の冒頭で「現在の最新仕様」を明記する。

### 2. 曖昧表現の具体値化

「○○のように」という曖昧表現をすべて具体値に置き換えてから提出する。

### 3. 1フェーズ1チャットの原則

1フェーズ1チャットの原則を維持。長いチャットは Claude Code の精度を下げる。

### 4. 位置指定の3点セット

指示書内で「変更対象箇所」を指定する際は、以下3点を併記する。

- 節番号（指示書内のセクション番号、例: §3-5）
- 実見出しテキスト（変更対象ファイル内の見出しを原文ママ、例: "## 運用背景"）
- 行番号（指示書作成時点の行番号、例: 行42-58）

1点だけ（節番号のみ・行番号のみ）だと、編集中に行番号がずれた時や見出しが
変更された時に齟齬が生じる。3点併記により、いずれか1点が変動しても残りの
2点で位置を一意化できる。

### 5. 変更前文言の完全一致確認

各 Edit 実行前に、指示書記載の「変更前文言」と実ファイルの該当箇所が
**1文字単位で完全一致** するかを確認する。

- 確認は GitHub MCP または `git show` 等で実ファイルから取得した原文と
  比較する
- 末尾改行・空白・全角半角・改行コードの差異も検査する
- 不一致を発見した場合、独自判断で Edit を進めず、太一さんに確認を求める
- 「微妙な差異だから合わせて Edit する」判断は禁止(指示書側の誤りなのか
  実ファイル側の差異なのかを Claude Code 単独で判定できない)

過去事例: 「不自然な否定文が実装直前に検出」事故。指示書の変更前文言が
実ファイルと微妙に異なっており、Edit 実行段階で初めて発覚した。

### 6. 基準SHA固定禁止

指示書内に基準SHA（main の SHA、変更対象ファイルの blob SHA 等）の
固定値を埋め込まない。代わりに作業開始時に Claude Code に動的確認させる。

- 指示書には `git rev-parse HEAD` 等のコマンドを記載する
- 確認した SHA は作業ログ・PR本文に記録する
- 指示書に固定値を埋め込むと、指示書作成時点と実装時点で SHA が変わった場合に
  「前提が壊れている」状態を検出できない

過去事例: 基準SHA固定で前提SHA衝突事故。指示書に固定 SHA を書いたまま、
実装時点で main が進んでおり、変更前文言の不一致が想定外の箇所で発生した。

### 7. 段階的ビルド検証（最低3回）

大規模変更（複数ファイル変更・複数サブタスク分割）では、作業途中で複数回
`npm run build` を実施する。

- タイミングは指示書側で明示する（例: 作業1完了直後・作業3完了直後・
  PR作成前）
- 最終ビルドだけだと、エラー発生箇所の特定が困難になる
- 大規模指示書では「途中で失敗 → 全体やり直し」が致命的なため、段階的
  検証で早期にエラー検出する

過去事例: 「途中失敗で全体やり直しが致命的」教訓。10ファイル分割の指示書で
最終ビルドだけ実施し、途中の作業3が原因のエラーが最後まで発覚せず、
作業4以降の修正コストが膨大になった。

### 8. Edit戦略: replace_all 禁止

`str_replace` を使う際、同一テキストが複数箇所に存在する場合でも
`replace_all` は使用しない。

- 各 Edit は **前後文脈を含めて一意化** された個別 Edit とする
- 同一テキストが複数箇所にある場合、それぞれの前後文脈を含めて識別する
- `replace_all` は「意図せぬ箇所まで変更される」リスクがあるため、
  指示書側でも禁止を明示する

替わりに、各箇所を「前後数行を含む文脈で一意化した old_str」で個別 Edit する。
これにより、Claude Code が変更箇所を取り違えるリスクを排除する。

### 9. コミット粒度の論理的分割

1コミット = 1論理単位の粒度を守る。論理単位の定義は指示書側で明示する。

- ファイル単位（1コミット = 1ファイル）が適切な場合
- 機能単位（1コミット = 1機能、複数ファイルに渡る）が適切な場合
- 領域単位（1コミット = 1領域、例: A領域→B領域→C領域 の3コミット構造）が
  適切な場合

指示書には「本タスクではどの粒度を採用するか」を明記し、Claude Code 側で
迷わない構造とする。粒度を明示しないと、Claude Code が独自判断で全変更を
1コミットにまとめてしまう、または逆に細分化しすぎる、といった事態が起きる。

### 10. コミットメッセージへの枝文脈併記

大規模指示書から派生したコミットでは、コミットメッセージに **枝文脈** を併記する。

- 例: "feat: GameScreen WindWidget pill実装（PR β §7.2 Buttons全面実装）"
- 枝文脈の書式: `（指示書名 §節番号 サブタスク名）`
- 後から `git log` で「このコミットがどの指示書のどのサブタスクから派生したか」を
  辿れるようにする

特に複数 PR から並行で派生したコミットが混在する場合、枝文脈なしでは
履歴の追跡が困難になる。

### 11. 大規模指示書の境界維持パターン

10ファイル分割等の大規模指示書では、各ファイル間の境界を明示する
チェーン構造を採用する。

- 各ファイル末尾に「次のファイルへ」のリンク・参照を記載
- 各ファイル冒頭に「前のファイルから引き継ぐ前提」を記載
- 途中で失敗した場合に、当該ファイルから再開可能な構造とする

書式テンプレートは現状未定。本タスク（4ファイル変更）では大規模分割は
不要だが、将来的に10ファイル超の指示書を作成する際は本項目に従う。

### 12. GitHub MCP 経由の実ファイル確認

GitHub MCP が利用可能な環境では、記憶ベースの引用ではなく、GitHub MCP
経由で実ファイルから取得した内容を優先する。

- 指示書作成時、変更対象ファイルの現状記述は GitHub MCP で取得する
- 取得した SHA は「確認時刻のスナップショット」として指示書に記載する
- 記憶ベースの引用は「ラベル体系・命名規則を実ファイルと取り違える」リスクが
  あるため、確実な実ファイル確認に置き換える

過去事例: 2026-04-19、known-pitfalls.md に存在しない 1-a〜1-d ラベル体系を
前提に指示書を作成した事案。記憶ベース記述は実害を生む。

並列作業時（複数チャット・複数 Claude AI が同じファイルを同時参照する場合）、
重複確認による無駄を避けるため、確認担当の分担を運用ルール側で整理する
（本項目では Claude Code 単独作業を前提とするため、分担問題は対象外）。
