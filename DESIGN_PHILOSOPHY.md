## 1. 製作背景と目的

### 1-1. アプリの概要

モルックスコアラー（molkky-scorer）は、フィンランド発祥のスキットルゲーム「モルック」専用のスコア記録PWA（Progressive Web App）である。技術スタックはReact + Viteで、単一ファイル構成（`src/App.jsx` が中心、`src/styles.css` が補助CSS、`api/analyze.js` と `api/sync.js` がVercelサーバーレス関数）を採用している。ソースコードはGitHub（`kimu2479jp-hue/molkky-scorer`）で管理され、Vercelで自動デプロイされる。本番URL は `molkky-scorer.vercel.app` である。

### 1-2. モルックという競技の特性とアプリに求められるもの

モルックは時間制限のある競技である。プレイヤーは自分の番が来るまでの短い待機時間中に、以下を同時に判断しなければならない。

- 自チームの現在スコアと50点までの残り点数
- 相手チームのスコアと上がり目（あと何点で50点に到達するか）
- 相手の調子やモメンタム（連続得点しているか、ミスが続いているか）
- ミス/フォルトの蓄積状況（3回連続ミスで失格）
- 37点ルールへの警戒（37点以上でフォルト→25点リセット）

これらを「限られた時間で」「遠目から」「直射日光下で」同時把握するという要件が、このアプリのUI設計を根本から規定している。一般的なスコア管理アプリとは要求水準が全く異なり、「屋外競技の審判端末」に近い思想で設計されている。

### 1-3. 使用環境の具体像

- **メイン表示端末**：iPadを三脚に固定して設置。プレイヤー全員（複数チーム・複数人）が遠目から画面を見てスコア状況を確認する。投擲位置から三脚までの距離は数メートルに及ぶことがある。
- **直射日光下での使用**が大前提。室内で綺麗に見えても、屋外で読めなければ意味がない。中間色、淡い色、半透明は日光で「飛ぶ」（見えなくなる）。
- **iPhoneでも運用する**が、iPadほどの画面サイズはないため、iPhone側ではスコア入力・簡単な確認程度の役割。
- **試合中の操作者は太一さん本人**（管理者）が多いが、メンバーが操作する場面もある。そのため、権限管理（管理者/メンバー）が実装されている。

### 1-4. 開発者の立場と開発経緯

太一さんが唯一の開発者兼管理者兼ユーザー。モルックのコミュニティで実際に使うための個人プロジェクトであり、商業製品ではない。

開発はClaudeとの対話を通じて進められており、以下の段階を経てきた。

1. **初期プロトタイプ（2026年2月末）**：Claudeとの最初のチャットで、React + Rechartsによるオフライン動作するモルックスコアツールの基本設計を決定。1〜12の得点ボタン、ミス/フォルトボタン、37点ルール、リアルタイムグラフ、途中参加機能を含む初版を生成。
2. **機能拡張期（2026年3月初旬）**：8つの大きな改善要望を一括処理。プルダウン見切れ修正、スタッツ画面の試合単位化、複数選択、スコア分布分析、設定画面UISwitch化、永続化（IndexedDB移行）、カレンダーUI、お気に入り99人上限などを実装。App.jsxは778行→1033行→さらに拡大。
3. **AI分析機能の追加（3月上旬）**：Anthropic APIを使ったプレイスタイルAI分析を実装。プロンプト調整を繰り返し、コメント文体を「流れる散文」に最適化。undefinedフィールドによるサーバーレスクラッシュの修正。
4. **クラウド同期の追加（3月上旬）**：Supabase（無料枠）を使ったデバイス間同期。sync_dataテーブル、api/sync.jsサーバーレス関数、3種類のマージ戦略を実装。
5. **権限管理システムの追加（3月上旬）**：2層PIN方式のadmin/member権限、同期コードマスク、AI分析のレート制限、設定画面の集約。
6. **iPhoneコピペ文字化け問題との格闘（3月上旬）**：iPhone Safari → GitHub Web UI のコピペで3,436箇所のスマートクォート置換、83箇所のUnicode三点リーダー混入が発生。App.jsxとapi/analyze.jsの両方に波及し、FUNCTION_INVOCATION_FAILEDの原因に。Python検証スクリプトで検出→全修正。
7. **Claude Codeへの移行（3月上旬〜）**：文字化け問題の根本対策として、GitHub Web UIでのファイル編集を廃止。Claude Code経由のgit push → PR → マージのワークフローに移行。
8. **UIリデザイン（3月上旬〜）**：ChatGPTの意見書を起点としたA案/B案比較を経て、アクティブ/非アクティブ方式によるUI全面刷新。配色、ヘッダーカード、アイコン、アニメーション、トランジションを含む大規模リデザインをClaude Code経由でpush。
9. **シャッフル演出のバグ修正と改善（3月8日）**：revealPos未定義による致命的クラッシュ（黒画面バグ）の修正、safety valve追加、3D回転演出、山札表現、発表画面簡素化。v3指示書＋追加修正の2ファイル体制でClaude Codeに渡す方式を確立。

### 1-5. 開発スタイルの基本ルール

- **実装前に必ず確認**：問題の説明 → 解決案の提示 → 太一さんの確認、というプロセスを必ず踏む。勝手に実装しない。
- **不明点は事前に列挙**：仮定で進めつつも、最後に「要確認」として質問を最大5個まで箇条書きでまとめ、太一さんの回答を得てから本実装に進む。
- **大規模な変更は段階的に分割して進める**：一度に全部を変えずに、Phase分けして安全な順番で実装する（影響範囲が小さく回帰リスクが低い修正から先に行う）。
- **ビルド検証は必ず行う**：コード変更後に `npx vite build` でビルドが通ることを確認してから出力する。

> **注記：出力文字数の制限について**
> Proプラン利用時（2026年3月第1週）、5時間あたり約45メッセージの制限下で、巨大なApp.jsx（1000行超）と複数の複雑な要望を同時処理する必要があったため、「1回の出力は最大1200字程度」という自主的な分割ルールを設けていた。これはProプランの処理能力制約への回避策であり、アプリの設計思想そのものではない。Maxプラン移行後はこの制限は不要であり、撤廃済みである。

## 2. UI設計思想（全画面共通）

### 2-1. 最上位の設計原則（GameScreen運用指針 v2）

画面ごとに設計の重心が異なるが、以下の優先順位は全画面に適用される普遍的な原則である。

**優先順位（この順序は絶対。下位が上位を覆すことは決してない）：**

1. **屋外で見やすい**（直射日光下で潰れない、コントラストが確保されている）
2. **今の操作対象が一瞬で分かる**（どのチームの番か、何をすべきか）
3. **誤認しにくい**（情報の取り違えが起きない構造的な保証）
4. **その上で見た目も整っている**（品位があるが、視認性を犠牲にした装飾はしない）

「スタイリッシュさ単独」を追求するデザインは明確に否定されている。目指すのは「おしゃれな管理画面」ではなく、「屋外競技で即読できる審判・入力端末」である。

### 2-2. 画面別の設計重心

GameScreen運用指針 v2 で定められた画面別方針は以下の通り。

| 画面 | 設計の重心 |
|------|-----------|
| **試合中画面（GameScreen）** | 視認性が最優先。すべての装飾は視認性に奉仕する |
| **スタッツ画面（StatsModal）** | おしゃれさを許容。情報の正確さと操作性が重要だが、磨き込みの余地がある |
| **設定画面（Settings）** | 操作のしやすさが最優先。美しさより迷わないこと |

### 2-3. 配色の基本方針 — A案（アクティブ/非アクティブ方式）の採用

ChatGPTが作成した意見書に基づくA案・B案の実機屋外テスト比較の結果、太一さんは**A案を最終採用**した。判断の根拠は「B案より読みやすかった」という実機テストの結果である。

#### A案の核心思想

従来の「チーム固有色」方式を完全に廃止し、「アクティブ/非アクティブ」の二項対立で全ての視覚情報を構成する。

- **チーム固有色は使わない**。全チームを常時別の派手な色で色分けしない。
- **アクティブチーム（現在投擲中のチーム）だけ**を強い色で強調表示する。
- **非アクティブチーム**はグレー系・無彩色寄り・低彩度色で抑制する。
- **視線が「今操作すべき対象」に自然に集まる**ことが最優先。
- 全チームを同じ強さで色分けしない。情報の重み付けは色の差で表現する。

#### 配色の具体ルール

| ルール | 理由 |
|--------|------|
| チームごとの常時派手色を廃止 | 瞬時にどのチームがアクティブか判別できなくなるため |
| 中間色を減らし、コントラストを高める | 直射日光下で中間色は飛んで消える |
| 半透明感・淡いグラデーション・弱い境界線を極力避ける | 日光下で視認不能になる |
| 罫線や区切りは必要に応じて強めにする | 情報の区切りが曖昧だと誤読する |
| 白地に濃い文字、または強い明暗差で読ませる | 日光下での最も堅実な可読性確保方法 |
| 白背景・明るい背景をベースにする | 意見書で「前向きに検討」と明記 |
| 文字色は埋もれない濃色を優先 | 色が近いと情報が区別できなくなる |
| 室内での美しさより日光下で潰れないことを優先 | 実使用環境が屋外であるため |

### 2-4. 情報ブロックの構成ルール

チーム名＋プレイヤー名＋スコア＋ミス数は、一つの視覚ユニット（情報ブロック）として統合表示する。これらの要素が画面上で散在してはならない。「チーム名はここ、スコアはあっち」というバラバラ配置は誤読の原因になる。

### 2-5. 背景のセミライト化

- **入力部（ScoreInput）は明るくする**。操作対象が明確に見えるようにする。
- **濃紺（ダークネイビー）はヘッダーとアクティブ要素にのみ使用**する。画面全体を暗くしない。
- 背景全体を均一なダークトーンにすると、アクティブ/非アクティブの差が出にくくなるため、セミライト方式で明暗差を作る。

### 2-6. 視線誘導の原則

視線誘導の主軸は**レイアウト・位置・サイズ**であり、色は補助手段にすぎない。

- 重要な情報は大きく、上部に配置する。
- 余白・整列・情報階層で品位を表現する。
- 色に頼りすぎない。色覚の多様性を考慮すると、位置とサイズで伝わる設計のほうが堅牢である。

### 2-7. 色数の制御

- 使用色は必要最低限に絞る。
- 「品位」はホワイトスペース、アラインメント、情報の階層構造、色数の制御で表現する。
- 多色使い（カラフルなチーム色など）はしない。色数が増えると、屋外で「全部同じに見える」問題が発生する。

## 3. 画面別の設計基準

### 3-1. 試合中画面（GameScreen）

#### 3-1-1. 設計哲学

視認性が絶対最優先の画面。試合中は操作に追われるため、0.5秒以内に必要な情報を読み取れなければ実用に耐えない。装飾は全て視認性に奉仕する。視認性を犠牲にする装飾は一切認めない。

#### 3-1-2. 情報の優先順位

画面上で最も目立つべき情報は、以下の順序で視覚的な重み付けをする。

1. 今何試合目・何ターン目か
2. 今どのチームの番か（アクティブチーム表示）
3. 現在の投擲プレイヤー名
4. 現在の合計点数
5. ミス/フォルトの残数や状態（3ミスで失格のため、ミス蓄積は命に関わる情報）

逆に、非アクティブチームの情報は「読めればよい」レベルであり、視覚的な主役にはしない。

#### 3-1-3. アクティブチーム表示の要件

「今このチームが操作対象」ということが**一瞬で**分かる必要がある。実現手段は太い帯、背景色の変更、ラベル、番号、囲み、左端バーなど方法を問わないが、以下の制約がある。

- 過剰に装飾するより、屋外での瞬間視認性を優先する。
- 色だけに頼らない。サイズ差や位置差でも分かるようにする。

#### 3-1-4. 入力部分の不変制約

以下は基本的に大きく変えない。根本的な作り変えは求めていない。

- **1〜12の円形ボタン配置**（3×4グリッド）：そのまま維持
- **決定ボタン**：維持
- **フォルトボタン**：維持
- **ミスボタン**：維持
- **ボタンサイズ**：現行の `isTablet` 判定ロジックの上限を超えない。iPad/iPhone両対応で安定性を維持するため。

改善の主要対象は「上部の見せ方」「色設計」「強調の仕方」である。

#### 3-1-5. スコア表

- スコア表そのものの基本構造は維持する。
- 試合結果画面から表示されるスコア表と同じ `ScoreTable` コンポーネントを共有する。
- チームヘッダー、チームカラー、プレイヤー名の縦書き、計列、ミス/フォルトの色分け等を含む。

#### 3-1-6. マルチチーム対応（2チーム・3チーム・4チーム）

この設計は特に重要であり、太一さんが意見書で詳細に言及している。

- **2チーム時の「VS」中心構図に依存しない**。VSを前提とした対戦演出構図は、3チーム・4チームでは構造的に破綻する。
- 3チーム以上では以下の方式を採用する：
  - アクティブチームだけを強調表示する
  - それ以外は横並びまたは一覧で非アクティブ表示する
- **重要なのは、対戦演出ではなく「今どこを見るべきか」が分かること**。
- 2チーム/3チーム/4チームそれぞれで、アクティブ/非アクティブの見せ方が破綻しないように設計する。

#### 3-1-7. iPad固有の数値設定（UIリデザイン後の確定値）

#### 3-1-8. ゲームルールの実装

- **37点ルール**：37点以上の状態でフォルト（モルッカーリを踏む）した場合、強制的に25点にリセットする。UIはフォルトボタン押下時に自動適用。
- **3ミス失格ルール**：連続3回ミスで失格。ミスドットで蓄積を視覚的に表示。
- **50点ぴったり勝利**：50点を超えた場合は25点にリセット（通常のオーバー処理）。ぴったり50点で上がり。

#### 3-1-9. 途中参加機能

試合中に急遽参加する人が現れた場合、人数が少ないチームに優先的に1人追加できる。お気に入りプルダウンから選択する方式。

#### 3-1-10. 進行中試合の永続化（クラッシュ耐性）

- **保存タイミング**：各ターン完了時にhistory/teamOrder/currentTurnをIndexedDB（`mk-game-progress`）に自動保存。
- **保証範囲**：強制終了時でも再起動すれば、少なくとも「試合途中の1試合前以前の全データ」は、削除ボタンで消さない限り永続保持される。
- **復元ダイアログ**：アプリ起動時に進行中データが検出されたら「再開する」/「破棄する」の選択ダイアログを表示。
- **復元粒度**：現在のターンまで完全再現する。途中のタイムスタンプは復元対象外。
- **正常終了時の挙動**：試合正常終了時や設定画面に戻った時に進行中データを自動削除する。

### 3-2. 試合結果画面（GameResult）

#### 3-2-1. スタッツ反映方針

試合終了時は**常にスタッツに自動反映する**。「累計スタッツに反映する/しない」のUI・機能は完全に撤廃済み。これは以前の設定画面UISwitch化の議論を経て最終決定された。

#### 3-2-2. スコア表モーダル

各試合アイテムの空きスペースに適度な大きさの「スコア表」ボタンを配置。押すとその試合のスコア表（閉じるボタン付き）がフルスクリーンモーダルで表示される。リプレイデータ（`saveReplay()` で保存されたチーム構造・全ヒストリー・投げ順）を使用して `ScoreTable` コンポーネントを描画する。

#### 3-2-3. 決着表示のルール

| 決着パターン | 表示テキスト |
|-------------|-------------|
| 50点ぴったりで上がった場合 | 「上がり者:（プレイヤー名）」 |
| 相手が全員失格した場合 | 「勝者:（プレイヤー名）」 |
| チームが2人以上の場合 | 上記に加えて「勝利チームのメンバー: ○○, ○○」行を追加 |

#### 3-2-4. ヘッダーのiOS対応

`env(safe-area-inset-top)` をヘッダーpaddingに適用し、iPhoneのノッチ/Dynamic Island領域を回避する。スタッツモーダルのヘッダーにも同様に適用済み。

### 3-3. スタッツ画面（StatsModal）

#### 3-3-1. 設計哲学

スタッツ画面はGameScreenとは異なり、美的な磨き込み（おしゃれさ）を許容する画面である。ただし情報の正確さと操作性は当然求められる。試合中のような0.5秒判断は不要だが、データの見やすさ・選択操作のスムーズさは重要。

#### 3-3-2. タブ構造

| タブ | 内容 |
|------|------|
| カレンダー | 本格カレンダーUIによる日付/期間での試合選択 |
| 直近の試合 | 直近30試合のリスト表示、チェックボックスで複数選択 |

「モルログプラス」タブは不要（競合アプリの機能であり、本アプリには不要と判断）。
既存の「今日/今週/今月/今年」期間ボタンは全て廃止。「累計」ボタンのみ残す。

#### 3-3-3. カレンダータブの詳細仕様

- **本格カレンダーUI**を実装する。シンプルな日付ピッカーや既存ライブラリの最小実装ではなく、競合アプリの画像参考レベルの本格版。
- **単一日付モード / 期間選択モード**の切替が可能。
- **試合がある日にマーカー表示**（カレンダー上のドットなど）。
- **年の操作**：
  - 年をタップすると年ピッカーを表示する。
  - 表示する年は「今年」のみ。年を越した時に、その時点の翌年が自動追加される仕様（2026年中は2026のみ。2027年になったら2027が追加）。
  - 年タップでその年の全試合をトグル（全選択⇔全解除）。**期間選択モードのみで動作**する。現在日時より未来は含まない。

  - 月をタップするとその月の全試合を一括選択/解除トグル。
  - 期間表示がその月の1日〜末日に自動設定される。
- **全期間の試合リスト一覧**からの複数選択に対応。

#### 3-3-4. 直近の試合タブの詳細仕様

- **表示件数**：直近30試合（旧20試合から拡大）。
- **ページネーション**：10件/ページ × 3ページ。ページ番号ボタンで切替。

  - 時刻
  - 上がりプレイヤー（50点決着時）/ 勝者（失格決着時）
  - 決着種目（50点決着 / 失格決着）

#### 3-3-5. レーダーチャート

#### 3-5-2. 致命的バグの修正履歴（教訓として記録）

ShuffleAnimationコンポーネントには、アプリ全体を操作不能にする致命的バグが存在した。

- **原因**：`revealPos` がファイル内で1箇所使用されていたが、どこにも定義されていなかった。`ReferenceError: revealPos is not defined` が発生し、コンポーネントがクラッシュ。しかし、overlayの `position:fixed` だけが画面に残り、操作不能の黒画面になった。
- **修正**：
  - `revealPos` を `useRef` で正しく定義
  - `stackOff2` の定義を `if(dealing)` ブロック外に移動（スコープバグ）
  - `cardDealTime` を `revealPos[idx]*perCard` で正しいタイミングに修正
  - **安全弁（safety valve）**：フェーズが8秒以上停滞した場合、自動で `onDone()` を呼んで安全に閉じる
  - return全体を try-catch でラップし、エラー発生時は `onDone()` を呼んで画面をロックしない

#### 3-5-3. 演出の詳細（v3指示書＋追加修正に基づく）

- **めくり演出**：CSS 3D transform による回転アニメーション。カードの裏面から表面に回転してチーム名を表示。
- **山札の視覚表現**：カードが積まれたように見せるスタック表現。
- **配り順のランダム化**：固定順ではなく、毎回ランダムな順序でカードを配る。
- **チーム発表画面**：シンプルに（チーム名＋メンバー名のみ）。マーク・番号は不要。
- **位置の調整**：山札と配られたカードの位置が重ならないようにする（マージ済み、軽微な位置調整）。

#### 3-5-4. Claude Codeへの指示書の渡し方（確立済みの運用ルール）

- **2ファイル体制**で渡す：メイン指示書（`claude-code-shuffle-fix-v2.md`）＋追加修正（`claude-code-shuffle-bugfix-addendum.md`）。
- **同時に渡す**（分けて渡すと手戻りが発生する）。
- **指示文の例**：「以下の2つの指示書を読んで作業してください。addendumの内容はメインより優先してください。最初に最優先バグ修正（safety valve）を行い、それが動くことを確認してから他の変更に進んでください。」

## 4. データ・ストレージ設計

### 4-1. ストレージ構成一覧

| データ種別 | 保存先 | キー / ストア名 | 容量上限 | 備考 |
|-----------|--------|----------------|---------|------|
| お気に入り | localStorage | `mk-fav` (primary) / `mk-fav-bk` (backup) | 99人 | 3段階フォールバック |
| スタッツ | IndexedDB | `mk-molkky-db` → `kv` ストア → key: `"stats"` | 100,000試合 | 超過時は最古から自動消去 |
| リプレイ | IndexedDB | `mk-molkky-db` → `kv` ストア → key: `"replays"` | 100,000試合 | 同上 |
| 進行中試合 | IndexedDB | `mk-game-progress` | 1試合分 | ターン完了ごとに上書き保存。正常終了時に自動削除 |
| AI分析キャッシュ | IndexedDB | `mk-molkky-db` → `analysis` ストア | — | 32日間の有効期限付き |
| 管理者PIN | localStorage | — | — | 端末ローカル。クラウド同期対象外 |

### 4-2. IndexedDB移行の経緯

当初はlocalStorageに全データを保存していたが、localStorageの容量制限（約5MB）では100,000試合の保存が不可能なため、IndexedDBに移行した。IndexedDBは数百MB〜GB級の容量があり、十分に対応できる。

- **IDB名**: `mk-molkky-db`
- **IDBバージョン**: 1 → 2（`analysis` オブジェクトストア追加時にバージョンアップ）
- **`onupgradeneeded` の注意点**: `db.objectStoreNames.contains()` チェックが必須。既存の `kv` ストアを壊さずに `analysis` ストアを追加するため。
- **移行処理**: 初回起動時にlocalStorageにデータがあればIndexedDBに移行し、localStorage側を削除。
- **インメモリキャッシュ**: `_cache` オブジェクトでstats/replaysをメモリ上に保持し、読み取りは同期的に行う。書き込みは非同期で永続化（fire-and-forget）。
- **超過時の自動消去**: `_trimOldestStats()` 関数で全プレイヤーの全レコードの日付を収集し、最古のものから削除。

### 4-3. クラウド同期（Supabase）

#### 4-3-1. アーキテクチャ

```
[iPad/iPhone] ← pushToServer/pullFromServer → [api/sync.js (Vercel Serverless)] ← REST API → [Supabase (Free Tier)]
```

#### 4-3-2. Supabaseテーブル

テーブル名: `sync_data`
- `sync_code TEXT PRIMARY KEY` — ユーザー定義の同期コード（デバイス識別子兼復元キー）
- `favorites JSONB` — お気に入りプレイヤー配列
- `stats JSONB` — プレイヤーごとのスタッツデータ
- `replays JSONB` — リプレイデータ
- `updated_at TIMESTAMP` — 最終更新日時
- RLS（Row Level Security）有効、anon keyによるpermissiveポリシー

#### 4-3-3. マージ戦略の最終決定と理由

| データ種別 | 戦略 | 理由 |
|-----------|------|------|
| favorites | **server-wins**（サーバー勝ち） | 管理者がお気に入りを削除した場合、その変更をメンバー端末にも確実に伝播させるため。unionマージだと管理者が消しても他端末から復活してしまう |
| stats | **union merge** | プレイヤーごとのゲームレコードを日付フィールドで重複排除しつつ結合。各端末で蓄積された試合履歴を全て保存するため |
| replays | **union merge** | キーベースの結合。同様に累積データの保存のため |

#### 4-3-4. 同期のトリガー

- **自動プッシュ**: `saveGameStatsToDB`, `saveReplay`, `saveFavsRaw`, `deleteStatsByPeriod` の呼び出し後、2秒のデバウンス付きで自動発火。
- **自動プル**: アプリ起動時に同期コードが存在する場合、自動的にサーバーからデータを取得。

#### 4-3-5. 環境変数

Vercelプロジェクト（`molkky-scorer` / チームslug: `kimu2479jp-hues-projects`）に以下の3つを設定する。

| 変数名 | 用途 |
|--------|------|
| `ANTHROPIC_API_KEY` | AI分析（api/analyze.js） |
| `SUPABASE_URL` | クラウド同期（api/sync.js） |
| `SUPABASE_ANON_KEY` | クラウド同期（Publishable key形式: `sb_publishable_...`） |

> **注意**: Supabase UIの変更により、anon keyの表示名が「anon public」から「Publishable key」に変更され、キー形式も `eyJhbG...` から `sb_publishable_...` に変わった。

#### 4-3-6. 複数Vercelプロジェクト問題

同一GitHubリポジトリに4つのVercelプロジェクトが接続されていた時期があった。`rj9r` は削除済み。現在のアクティブプロジェクトは `molkky-scorer`（`molkky-scorer.vercel.app`）のみ。環境変数はこのアクティブプロジェクトにのみ設定する。

## 5. AI分析機能（プレイスタイルAI分析）

### 5-1. 概要

Anthropic APIを利用したプレイヤー分析機能。Vercelサーバーレス関数 `api/analyze.js` がフロントエンドとAnthropic API の間を中継する。

### 5-2. 名称と動作方式

- **旧名**: 「得点スタイル」
- **新名**: 「プレイスタイルAI分析」
- **発火方式**: 手動トリガーのみ。画面を開いた時やレンジ変更時の自動発火は完全に撤廃した。
  - プレイヤーごとの個別分析ボタン
  - 一括の「全員分析」ボタン
- **分析対象**: お気に入り登録されたプレイヤーのみ。

### 5-3. 分析に使用するスタッツ項目

APIに送信するプレイヤー情報は以下の通り。

- 試合数（gameCount）
- 勝率（winRate）
- ミス率（missRate）— ミスした投擲の割合
- 上がり決定率（finishRate）— 38点以上から50点丁度を1投で決めた割合
- 投擲平均点（avgPts）
- ブレイク平均（breakAvg）— 先攻時の初投の平均得点
- お邪魔成功率（ojamaRate）— 相手の上がり目を妨害した割合
- お邪魔試行数（ojamaAttempts）
- 2ミス後リカバリ平均（recAvg）
- 先攻勝率（firstWinRate）— nullの場合あり
- 後攻勝率（lastWinRate）— nullの場合あり

### 5-4. コメント文体の要件（プロンプトで厳密に指定）

| 要件 | 詳細 |
|------|------|
| 文章形式 | 流れる散文（メモ形式）。3〜4行、合計50〜85文字 |
| 行の扱い | 1文が行を跨いでよい |
| 表現スタイル | 「○○タイプ」「○○型」で終わらせず、物語を感じさせる表現 |
| 弱点の扱い | 弱点にも触れるが、愛嬌ある言い換えで表現する |
| 修辞技法 | 比喩・擬人化・意外な例えを積極的に使う |
| 数値 | アラビア数字（1, 2, 50%など） |
| 漢字 | 慣用句や固有表現（二面性、一歩など）は漢字のまま |

太一さんが書いた「ミラクル」の例文のスタイルをプロンプトの参考例として組み込み済み。

### 5-5. 技術的な安全策

#### サーバーサイド（api/analyze.js）

- `n()`, `pct()`, `pt()` の安全な数値変換ヘルパー関数を導入。undefinedフィールドが渡されても `.toFixed()` でクラッシュしない。
- Anthropic APIのレスポンスをまずrawテキストとして取得し、その後JSON.parseする二段階パース（エラー時に意味のあるメッセージを返すため）。

#### フロントエンド側

- 全数値フィールドに `||0` のデフォルト値を設定。
- `_analysisPending` Setで同一プレイヤーへの並行リクエストを重複排除。
- `useEffect` のクリーンアップで `cancelled=true` がAPIレスポンス受信前に発火する問題を、`active` フラグの分離で解決（無限fetch-cancelループの根本原因だった）。
- エラー状態を明示的にUIに表示（以前はエラーが完全に隠蔽されていた）。

#### APIモデル

- `claude-sonnet-4-5-20250929` を使用（旧モデル名 `claude-sonnet-4-20250514` は非推奨に変更済み）。

### 5-6. クレジット管理

- Anthropic APIには残高照会エンドポイントが存在しないため、累積使用量カウンター方式（案B）を採用。
- Anthropicコンソールへのリンクは管理者にのみ表示する。

### 5-7. キャッシュ

- IndexedDBの `analysis` オブジェクトストアに保存。
- 有効期限は32日間。期限切れのキャッシュは無視される。
- IDBバージョンを1→2に上げて `analysis` ストアを追加する際、`onupgradeneeded` で `db.objectStoreNames.contains("kv")` チェックが必須（既存 `kv` ストアを壊さないため）。

## 6. 端末別の考慮事項

### 6-1. iPad — メイン共有表示端末

- **用途**: 三脚に固定し、屋外で全プレイヤーが遠目から見るメイン表示端末。スコア入力もiPadから行う。
- **`isTablet` 判定**: フォントサイズ・ボタンサイズ・余白を拡大するフラグ。入力エリアのボタンはこのロジックの上限を超えない。
- **maxWidth制限の撤廃**: 以前はスコア入力エリアにmaxWidthが設定されていたが、iPadの広い画面で不要な余白が生まれるため撤廃。代わりに `padding: 0 40px` で左右余白を制御。

#### 6-2-1. iPhoneコピペ文字化け問題（最大の技術的障害の記録）

iPhone Safari → GitHub Web UI でコードをコピペした際に発生した大規模文字化けは、プロジェクト史上最大の技術的障害だった。

- **スマートクォート置換**: `"` → `"` / `"` が3,436箇所で発生。JavaScriptの文字列リテラルが全て壊れた。
- **Unicode三点リーダー**: `...`（スプレッド構文）→ `…`（U+2026）が83箇所で発生。Reactの状態更新コードが全滅。
- **enダッシュ**: `--`（デクリメント演算子）→ `–`（U+2013）に置換。ループカウンターが壊れた。
- **アスタリスク**: `_`（変数名のアンダースコア）→ `*` に置換。
- **Markdownコードブロックマーカー**: ` ``` ` が本文中に混入。

この問題はApp.jsxだけでなくapi/analyze.jsにも波及し、`FUNCTION_INVOCATION_FAILED` エラーの原因にもなった。Python検証スクリプト（`content.count('\u201c')` 等）で検出し全修正。

**根本対策**: GitHub Web UIでのファイル編集を廃止し、Claude Code経由のgit pushワークフローに移行。これにより文字化け問題は完全に解消された。

#### 6-2-2. iOS Safari固有の対策一覧

| 問題 | 対策 | CSS/JS |
|------|------|--------|
| ダブルタップズーム（チップ拡大バグ） | `touch-action: manipulation` | 全インタラクティブ要素に適用 |
| ボタン長押し時のテキスト選択 | `-webkit-user-select: none` / `-webkit-touch-callout: none` | button, [role="button"], .no-select |
| contextmenu（長押しメニュー） | `contextmenu` イベントの `preventDefault()` | input/textarea以外の全要素 |
| オーバースクロール（黒背景露出） | `overscroll-behavior: none` | html, body, 全fixedモーダル, GameScreen, SetupScreen |
| body背景とモーダル背景の色差 | body背景を `#0b1526` → `#14365a` に統一 | styles.css |
| ノッチ/Dynamic Island領域の干渉 | `env(safe-area-inset-top)` | ヘッダーpaddingに適用 |
| viewport拡大 | `maximum-scale=1, user-scalable=no` | index.html viewport meta（設定済み） |

## 付録: 技術的な教訓と知見

### A. iPhoneコピペ検出の確実な方法

`grep` ベースのアプローチではUnicodeバリアントを見逃す。Python スクリプトで `content.count('\u201c')` 等を使うのが確実。

### B. GitHub API のレート制限

`https://api.github.com/repos/[user]/[repo]/git/trees/main?recursive=1` は認証なしではレート制限に頻繁にヒットする。ファイル内容の取得には `curl -sL https://raw.githubusercontent.com/[user]/[repo]/main/[path]` を直接使うほうが確実。

### C. Vite ビルドの注意点

`@supabase/supabase-js` はサーバーレス関数（api/）でのみ使用しているが、package.json の dependencies に含めないとビルドがインポート検証で失敗する。

### D. Anthropic APIモデルの非推奨化

`claude-sonnet-4-20250514` → `claude-sonnet-4-5-20250929` への移行が必要だった。モデル名変更はapi/analyze.jsの500エラーの原因の一つだった。

### E. React hookのキャンセルパターン

`useEffect` のクリーンアップで `cancelled=true` を設定するパターンは、非同期API呼び出しのキャッシュ書き込みを阻害する場合がある。`active` フラグをキャッシュ書き込みとキャンセルチェックで分離することで、無限fetch-cancelループを防止できる。
